# 产品需求文档（PRD）- SwarmAI Platform

版本：v0.1（Demo）

## 1. 背景与目标

### 1.1 背景

无人系统（无人机/无人车/无人船/固定点位）逐渐走向“多平台异构、任务多样、链路受限、现场不确定”的应用形态。传统方案常出现：

- 任务方案依赖人工指挥 + 固定脚本/状态机，难以适应突发情况
- 数据链/消息协议由通信与软件工程师手工维护，迭代成本高
- 现实部署环境常为“离线/内网”，无法直接调用云端大模型
- 缺乏闭环评测：方案是否好、协议是否可用、链路是否稳，缺少可量化指标

### 1.2 产品目标（MVP）

MVP 目标是构建一个可商用交付的**软件平台**，实现：

1. 任务需求输入（自然语言/结构化 JSON）
2. 自动输出：分层决策方案（decision_json）+ 消息/协议定义（protocol_json）+ 编码/解码骨架（C/Python 等）
3. 自动联动仿真/数字孪生进行验证与指标评测（coverage、时延、丢包下鲁棒性等）
4. 支持异构设备接入：通过“统一控制平面协议 + 适配器网关层”对接 MAVLink/ROS2/MQTT/厂商 SDK

## 2. 目标用户与场景

### 2.1 目标用户

- 方案/系统工程师：做方案、做体系结构、做演训推演
- 通信/软件工程师：做协议设计、写编码解码、做联调
- 仿真/测试工程师：搭建仿真场景、跑压力测试、做指标评测
- 一线业务人员：巡检、搜救、安防巡逻、演训游戏化等（民用场景）

### 2.2 典型场景（优先级从高到低）

1. 仿真演训/对抗（游戏化）- 快速迭代规则与协议，验证闭环
2. 巡检/巡逻/搜救 - 多平台协同与链路受限下的信息汇聚
3. 物资投送/应急广播等载荷协同（以“载荷插件”形式实现，不在 MVP 细化）

## 3. 产品范围

### 3.1 MVP 功能清单

- 任务输入：支持 mission_json + scene_json
- RAG 知识库：支持协议字段/历史案例检索
- 决策生成：默认规则引擎，预留 LLM/NanoGPT-Edge 插拔
- 协议生成：输出 protocol_json + 代码骨架
- 仿真闭环：轻量 2D 仿真 + 指标输出（JSON 报告）
- API：REST 接口（/v1/plan, /v1/sim/run）

### 3.2 非目标（MVP 不做）

- 高保真飞行动力学/物理仿真（交给 PX4/Gazebo 等）
- 真实设备自动驾驶控制细节（交给厂商 SDK/MAVLink/ROS2）
- 带武器/危险用途实现（本平台禁止）

## 4. 关键需求（需求条目）

### 4.1 功能需求

- FR-01：输入 mission_json + scene_json，生成 decision_json（可校验 schema）
- FR-02：结合 KB 检索结果，对协议/案例进行检索增强
- FR-03：生成 protocol_json（包含消息头、消息类型、字段、方向、触发条件）
- FR-04：根据 protocol_json 生成代码骨架（C 结构体 + Python dataclasses）
- FR-05：将 decision/protocol 导入仿真，输出指标报告
- FR-06：提供 API 服务，支持其他系统集成

### 4.2 非功能需求

- NFR-01：离线可部署（内网环境）
- NFR-02：可审计（decision_json 是规则级、可读、可版本化）
- NFR-03：可扩展（适配器/载荷插件/协议版本管理）
- NFR-04：可评测（每次运行可产出固定格式评测报告）

## 5. 成功指标（MVP 验收）

- 端到端闭环一次跑通：计划生成 + 协议生成 + 代码骨架 + 仿真 + 指标输出
- 在 10 分钟仿真中达到覆盖率 >= 0.7（Demo 指标，可调整）
- 在丢包率 10% 下仍能完成“目标上报 -> GS 响应 -> 簇首重分配”的消息闭环

## 6. 版本规划（建议）

- v0.1：规则引擎闭环 + 基础协议/代码生成 + 2D 仿真
- v0.2：接入 Teacher LLM（在线/强能力）生成更多样方案，完善 schema 校验
- v0.3：训练 Student（NanoGPT-Edge 或 LoRA 学生模型）离线部署
- v0.4：对接 PX4/ROS2/Gazebo，形成更高保真仿真闭环
- v1.0：多租户、权限、审计、运维、商业交付套件
